# Pi Python Agent Runtime SDK 实施计划

## Summary

做一个给 Python 后端用的 **持久化 Agent 运行器**。它不是新的 Agent 框架，而是让 Python 服务把任务交给 pi Agent，并负责运行、隔离、保存现场、恢复会话和交付结果。

核心定位：

> Python 服务提交一个 Agent Job，SDK 负责让它稳定跑完，并保存可恢复的工作现场。

第一版先服务单 Agent 任务，后续再扩展到任务编排和多 Agent 协作。

## Key Changes

### 公共接口

SDK 第一版提供这些核心对象和接口：

- `AgentSpec`：定义 Agent 名称、系统提示词、技能包、工具、模型配置。
- `AgentJob`：一次 Agent 任务，包含输入、状态、会话、工作区、产物。
- `AgentRuntime`：负责启动、停止、恢复 Agent 运行。
- `WorkspaceStore`：负责保存和加载工作区。
- `SessionStore`：负责保存和恢复会话历史。
- `ArtifactStore`：负责保存任务产物。
- `JobManager`：负责创建、查询、停止、恢复任务。

最小 API：

- `create_job(spec, input, workspace)`
- `stream(job_id)`
- `stop(job_id)`
- `resume(job_id)`
- `get_status(job_id)`
- `list_artifacts(job_id)`
- `download_artifact(job_id, path)`

默认实现：

- 本地目录存储。
- pi RPC 本地进程运行。
- 单 Agent、单任务工作区。
- FastAPI 示例。

## Implementation Phases

### Phase 1：本地单 Agent MVP

状态：已完成（2026-06-05）

目标：Python 服务能在本机提交一个 Agent 任务并拿到结果。

实施边界：

- 使用 pi RPC 作为底层运行方式。
- 支持创建任务、发送输入、流式接收输出、停止任务。
- 每个任务有独立工作区。
- 会话历史保存到本地。
- 支持传入模型配置、环境变量、技能包路径、工具配置。
- 提供 FastAPI 示例。

验收标准：

- FastAPI 可以创建任务并返回 `job_id`。
- 可以实时看到 Agent 输出。
- 任务结束后可以查看最终结果。
- 工作区里能看到 Agent 生成或修改的文件。
- 停止任务后，会话和工作区不损坏。
- Python 服务重启后，还能查询历史任务状态。

### Phase 2：持久化和恢复

状态：已完成（2026-06-05）

目标：任务现场可以跨服务重启恢复。

实施边界：

- 任务状态、会话历史、工作区、产物分开保存。
- 支持本地目录和 S3/MinIO。
- 支持恢复指定任务。
- 支持导出一次任务的完整记录和产物。
- 明确失败状态，避免半残任务看起来像成功。

验收标准：

- 杀掉 Python 服务后重启，历史任务不丢。
- 会话能继续，不需要从头开始。
- 工作区能恢复到上次状态。
- 对象存储不可用时，任务进入明确失败状态。
- 用户能下载任务产物和运行记录。

### Phase 3：生产运行边界

状态：已完成（2026-06-05）

目标：让 Agent 可以安全跑在普通服务器和 Kubernetes。

实施边界：

- 普通服务器支持本地进程和 Docker。
- Kubernetes 支持 Pod + PVC。
- 支持超时、资源限制、环境变量注入。
- 支持日志、错误原因、退出状态。
- 支持任务清理，避免遗留无用工作区。
- 不做复杂集群调度。

验收标准：

- 同一个任务能在本地、Docker、Kubernetes 跑通。
- 每个任务的工作区互相隔离。
- Agent 崩溃后，Python 服务能拿到明确失败原因。
- 超时任务能自动停止。
- PVC 里能看到会话和工作产物。
- 密钥不会出现在日志和任务记录里。

### Phase 4：任务接口和轻量编排入口

目标：用户可以管理多个 Agent 任务，但不依赖外部 Agent 框架。

实施边界：

- 提供统一的任务创建、查询、停止、恢复接口。
- 支持任务依赖，比如 B 读取 A 的产物后运行。
- 支持任务状态、日志、产物、失败原因统一查询。
- 支持用户在自己的 Python 服务里写流程。
- 不把 LangGraph、OpenAI Agents SDK、CrewAI 示例作为主线。
- 不做可视化流程图。

验收标准：

- 用户能提交多个任务。
- 一个任务能读取另一个任务的产物。
- 前置任务失败后，后续任务不会误跑。
- 每个任务的输入、输出、状态都能追踪。
- 不依赖外部 Agent 框架，也能跑简单多步骤流程。

### Phase 5：多 Agent 协作

目标：支持多个 Agent 围绕同一个目标协作。

实施边界：

- 支持多个 Agent 使用同一个项目工作区。
- 支持串行、并行、人工确认后继续。
- 支持任务之间传递上下文和产物。
- 支持失败重试。
- 支持恢复到某个中间状态继续。
- 编排能力只做和 Agent 运行强相关的部分，不扩展成通用工作流平台。

验收标准：

- 多个 Agent 可以处理同一个目标。
- 不同 Agent 的操作记录清楚可查。
- 一个 Agent 失败不会污染其他任务。
- 可以从中间步骤恢复继续。
- 跑通真实场景：分析代码库 -> 修改文件 -> 生成报告 -> 人工确认。

### Phase 6：产品化和团队使用

目标：让团队能长期使用和维护。

实施边界：

- 提供任务列表、运行记录、产物查看。
- 提供耗时、成本、失败率统计。
- 提供权限、密钥隔离、审计记录。
- 提供部署模板和升级说明。
- 建立稳定版本发布流程。

验收标准：

- 团队能查看所有任务状态。
- 能追踪一次任务花了多少时间和成本。
- 密钥不会暴露在任务记录里。
- 出问题能根据日志定位。
- SDK 升级不破坏已有会话和工作区。

## Test Plan

每个阶段都必须覆盖这些测试：

- 正常任务：提交任务、流式输出、完成、查看产物。
- 停止任务：中途停止后，状态正确，现场不损坏。
- 恢复任务：服务重启后能恢复会话和工作区。
- 失败任务：pi 崩溃、模型失败、存储失败时状态明确。
- 隔离测试：两个任务不能互相污染工作区。
- 存储测试：本地存储、S3/MinIO、PVC 各自跑通。
- 安全测试：环境变量和密钥不出现在日志、产物、会话记录里。
- 示例测试：FastAPI 示例、本地运行示例、Kubernetes 示例都能按文档跑通。

## Assumptions

- 底层 Agent 能力优先复用 pi，不重写 Agent 内核。
- 第一版只做单 Agent，不做复杂多 Agent。
- 第一版默认支持本地目录和 S3/MinIO。
- Kubernetes 支持从 Phase 3 开始进入主线。
- LangGraph、OpenAI Agents SDK、CrewAI、AutoGen 不作为核心依赖。
- 所有新功能都用一个标准判断：是否让 Agent 更容易启动、隔离、恢复、保存结果。
